# Задача 1330: Фикс повторного открытия главного меню на мобильном

**Дата:** 2026-01-27  
**Задача:** 1330  
**Тип:** Исправление ошибки, UI/UX, Мобильное меню

---

## Контекст

**Проблема:** При повторном открытии главного меню на мобильном устройстве возникали проблемы: меню могло открываться несколько раз одновременно, состояние меню не сбрасывалось корректно при закрытии, или класс `active` не удалялся с контейнера меню при закрытии панели.

**Ограничения:** 
- Мобильное меню использует библиотеку CupertinoPane для отображения панели
- При первом открытии создается клон контейнера меню (`mobileMenuContainer`) и инициализируется CupertinoPane
- При повторных открытиях используется уже созданный `mobileMenu` и показывается панель
- Необходимо обеспечить корректное управление состоянием меню при открытии и закрытии

**Базовое поведение:** 
- Метод `openMobileMenu()` вызывается при клике на кнопку открытия мобильного меню
- При первом открытии создается `mobileMenu` и `mobileMenuContainer`
- При повторных открытиях используется существующий `mobileMenu` и вызывается `setStartScreen()` для сброса состояния
- Класс `menu.container.active` добавляется к `mobileMenuContainer` при открытии

---

## Ключевые решения

### Управление состоянием мобильного меню при закрытии

**Решение:** Добавить обработку события `onWillDismiss` для корректного сброса состояния меню при закрытии панели: удаление класса `active` с `mobileMenuContainer` и сброс флага `isMobileOpened`.

**Обоснование:** 
- При закрытии панели через свайп или другие способы класс `active` оставался на `mobileMenuContainer`, что могло вызывать проблемы при следующем открытии
- Флаг `isMobileOpened` не сбрасывался при закрытии, что могло приводить к некорректному поведению при повторных открытиях
- Обработчик `onWillDismiss` вызывается перед закрытием панели, что позволяет корректно очистить состояние

**Влияние:** Обеспечивает корректное управление состоянием меню и предотвращает проблемы при повторных открытиях.

## Code Touchpoints

### `app/local/changes/template/src/js/app/AppHeaderMenu.js`

**Метод:** `openMobileMenu()` (строки 223-281)
- При первом открытии (`if (!menu.mobileMenu)`) создается контейнер, клон меню и инициализируется CupertinoPane
- В обработчике `onWillDismiss` (строки 264-266) добавлена логика для сброса состояния:

  - Сброс флага `isMobileOpened`: `menu.isMobileOpened = false`

  - Вызывается `setStartScreen()` для сброса состояния на начальный экран
  - Добавляется класс `active` к `mobileMenuContainer`
  - Вызывается `present()` для показа панели

**Метод:** `hideMenu()` (строки 305-331)
- При закрытии меню также вызывается `mobileMenu.hide()` для мобильной версии (строка 327)

## Gotchas (Pitfalls)

1. **Обработчик `onWillDismiss` вызывается перед закрытием**: Важно сбрасывать состояние именно в этом обработчике, а не в `onDidDismiss`, чтобы состояние было очищено до полного закрытия панели.

2. **Класс `menu.container.active`**: Это строка из `activeClasses.container`, которая равна `'opened'`. Необходимо использовать `menu.container.active` для доступа к значению класса.

3. **Флаг `isMobileOpened`**: Должен синхронизироваться с фактическим состоянием панели CupertinoPane. При открытии устанавливается в `true`, при закрытии - в `false`.

4. **Синхронизация с `isOpened`**: Флаг `isOpened` используется для десктопного меню, а `isMobileOpened` - для мобильного. Они должны быть независимы, но синхронизированы с фактическим состоянием соответствующих меню.

5. **Множественные вызовы `present()`**: CupertinoPane может корректно обрабатывать множественные вызовы `present()`, но лучше проверять состояние перед открытием, чтобы избежать конфликтов.

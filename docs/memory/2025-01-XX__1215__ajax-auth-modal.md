# Task Memory Card

Date: 2025-01-XX
Task Key: 1215
Title: AJAX загрузка модального окна авторизации

## Context
- Проблема: Модальное окно авторизации статически загружалось в DOM на каждой странице, увеличивая начальный размер страницы и замедляя её загрузку. Кнопка "Оформить заказ" в корзине использовала кастомный механизм открытия модального окна через `data-bs-toggle="modal"`, который не работал с динамически загружаемыми модальными окнами.
- Базовое поведение: Компонент `app:system.auth.full` статически включался в `header.php`, модальное окно всегда присутствовало в DOM. Кнопка в корзине использовала `data-bs-toggle="modal" data-bs-target="#login-or-register"` для открытия модального окна.

## Key Decisions
- Решение: Переход на AJAX загрузку модального окна авторизации через стандартный механизм `[data-toggle=ajax-modal]` → Унификация механизма открытия модальных окон, упрощение поддержки кода, улучшение производительности за счёт ленивой загрузки.
- Создание компонента `app:system.auth.link` → Вынос логики отображения ссылки на авторизацию в отдельный переиспользуемый компонент, упрощение `header.php`.
- Добавление поддержки выбора типа лоадера через data-атрибут `data-loader-type` → Гибкость в выборе типа лоадера (глобальный/на кнопке) в зависимости от контекста использования.

## Code Touchpoints
- `app/api/modal/auth.php` → Новый API endpoint для динамической загрузки модального окна авторизации через AJAX.
- `app/local/components/app/system.auth.link/` → Новый компонент для отображения ссылки на авторизацию/личный кабинет с поддержкой модального окна или прямой ссылки.
- `app/local/templates/trimiata/js/App.js` → Обновлён обработчик `[data-toggle=ajax-modal]` для поддержки `data-loader-type`, `data-add-to-cache`, `data-cache-type`. Упрощён обработчик `[data-role=go-to-checkout]` для работы только со ссылками.
- `app/local/templates/trimiata/js/AppApi.js` → Добавлена поддержка параметров `addToCache` и `cacheType` в метод `loadAjaxModal`.
- `app/local/templates/trimiata/js/AppAuthorization.js` → Добавлена проверка на дублирование обработчика перехода на checkout после закрытия модального окна в корзине.
- `app/local/templates/trimiata/header.php` → Заменён статический вызов компонента авторизации на компонент `app:system.auth.link`.
- `app/local/templates/trimiata/components/bitrix/sale.basket.basket/main/template.php` → Кнопка "Оформить заказ" для неавторизованных пользователей использует `[data-toggle=ajax-modal]` вместо `data-bs-toggle="modal"`.

## Gotchas (Pitfalls)
- Обработчик перехода на checkout после закрытия модального окна в корзине может дублироваться, если модальное окно уже загружено в кеше. Решение: проверка через атрибут `data-cart-checkout-handler` и использование `{ once: true }`.
- JS логика внутри модального окна авторизации (`AppAuthorization`) должна инициализироваться после загрузки HTML через AJAX. Решение: инициализация в `loadAjaxModal` после добавления модального окна в DOM.
- При использовании глобального лоадера событие `_loaded` должно диспатчиться на `document`, а не на кнопку. Решение: условная логика в обработчике `[data-toggle=ajax-modal]`.

## Verification
- Проверить, что модальное окно авторизации загружается через AJAX при клике на кнопку входа в header.
- Проверить, что кнопка "Оформить заказ" в корзине для неавторизованных пользователей открывает модальное окно авторизации.
- Проверить, что после успешной авторизации в корзине происходит автоматический переход на страницу оформления заказа.
- Проверить, что глобальный лоадер показывается при загрузке модального окна авторизации.
- Проверить, что модальное окно кешируется и не загружается повторно при повторном открытии.

## Follow-ups
- Возможно, стоит добавить поддержку других типов лоадеров (например, inline) через data-атрибут `data-loader-type`.
- Рассмотреть возможность добавления поддержки кеширования через service worker для модальных окон с `addToCache="true"`.

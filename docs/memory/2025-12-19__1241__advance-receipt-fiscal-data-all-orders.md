# 1241: Фискальные данные авансовых чеков для всех заказов: Тест

**Date**: 2025-12-19  
**Status**: Completed

## Context
Требовалось обеспечить передачу фискальных данных авансовых чеков для всех типов заказов в 1С, а не только для заказов с сертификатами. Ранее обработчик события печати чека работал только для заказов с сертификатами, что не позволяло передавать фискальные данные для обычных заказов с авансовыми платежами.

## Key Decisions
- Обработка всех авансовых чеков в событии `OnPrintableCheckSend` → метод `addReceipt` обрабатывает все чеки типа `advancepayment` → унификация обработки чеков для всех типов заказов → упрощение поддержки кода и гарантия единообразия
- Определение типа чека через проверку `isCertificateOrder()` → параметр `type` ('certificate' или 'advance') → корректная маршрутизация в методе `createReceipt` → правильная обработка данных в зависимости от типа заказа
- Перенос логики отправки оплаты из `OnSaleOrderPaid` в `onAfterCreateReceiptSend` → выполнение после получения ответа от 1С → согласованность данных и исключение дублирования операций → корректная последовательность обработки заказа
- Структура данных `Receipt`: добавление поля `payment` и типа в `certificate` → поддержка как сертификатов ('gift'), так и авансовых платежей ('online_payment') → гибкая структура для разных типов чеков
- Разделение API: `sell-certificate` в v2, список сертификатов в v1 → версионирование API → возможность изменений без влияния на существующие интеграции

## Code Touchpoints
- `app/local/php_interface/lib/events/sale/Order.php`
  - Метод `addReceipt()` (переименован из `addCertificateReceipt`) обрабатывает все авансовые чеки (advancepayment), определяет тип ('certificate' или 'advance') и добавляет задачу в очередь обмена с 1С
  - Удален метод `sendOrderPayment()` и его регистрация в событии `OnSaleOrderPaid` - логика перенесена в `onAfterCreateReceiptSend`
- `app/local/php_interface/lib/app/Exchange/Trimiata1c/Trimiata1c.php`
  - Метод `createReceipt()` теперь обрабатывает как сертификаты, так и авансовые платежи:
    - Для сертификатов: сумма из данных сертификата, тип 'gift', платежная система 'certificate'
    - Для авансовых платежей: сумма из заказа (`getPrice()`), тип 'online_payment', платежная система из заказа
  - Метод `onAfterCreateReceiptSend()` обрабатывает ответы от 1С для обоих типов:
    - Для сертификатов: сохранение номера и купона сертификата в свойства корзины, создание купона, изменение статуса
    - Для авансовых платежей: сохранение купона сертификата в свойство заказа `ORDER_ADVANCE_CERTIFICATE_PROP_CODE`, отправка оплаты заказа, установка статуса "в обработке"
  - Изменена структура API: метод `sell-certificate` перенесен в `certificates/v2/`, список сертификатов остался в `certificates/v1/`
- `app/local/php_interface/lib/app/Exchange/Trimiata1c/DataTypes.php`
  - В структуру `Receipt` добавлено поле `payment` (массив с ключом 'name') для передачи данных платежной системы (в коде используется `$_receipt1c->payment`, в структуре может быть поле `name`)
  - Обновлена документация поля `certificate`: теперь содержит сумму и тип ('gift' или 'online_payment')
- `app/local/php_interface/lib/app/Exchange/Trimiata1c/Helper.php`
  - В метод `getPaymentData()` добавлена логика получения номера авансового сертификата из свойства заказа `ORDER_ADVANCE_CERTIFICATE_PROP_CODE`
- `app/local/php_interface/lib/app/Order/Helper.php`
  - Добавлена константа `ORDER_ADVANCE_CERTIFICATE_PROP_CODE = 'ADVANCE_CERTIFICATE'` для хранения купона авансового сертификата

## Gotchas (Pitfalls)
- Логика отправки оплаты заказа теперь выполняется после получения ответа от 1С → если 1С недоступна или возвращает ошибку, оплата заказа не будет отправлена → необходимо контролировать статусы задач в очереди обмена и обрабатывать ошибки
- Метод `createReceipt` требует наличия фискальных данных в параметрах → если фискальные данные не были переданы в событии печати чека, метод не сможет сформировать корректный запрос → необходимо убедиться, что все необходимые данные присутствуют в `LINK_PARAMS` чека
- Для авансовых платежей используется сумма заказа (`getPrice()`) → если в заказе были применены скидки или изменения цены после печати чека, сумма может не соответствовать сумме в чеке → необходимо контролировать согласованность данных
- Сохранение купона сертификата в свойство заказа происходит только при наличии ответа от 1С с полем `certificate_coupon` → если 1С не вернула купон, свойство не будет заполнено → необходимо убедиться, что 1С всегда возвращает купон для авансовых платежей
- Метод `onAfterCreateReceiptSend` обрабатывает отправку оплаты только для онлайн-платежных систем и не-сертификатных заказов → заказы с другими способами оплаты могут требовать отдельной обработки → необходимо проверить все сценарии оплаты

## Verification
- Проверить, что все авансовые чеки (advancepayment) корректно обрабатываются событием `OnPrintableCheckSend`
- Убедиться, что фискальные данные передаются в 1С для всех типов заказов (сертификаты и обычные заказы)
- Проверить, что для авансовых платежей корректно заполняются сумма, тип и платежная система
- Убедиться, что купон сертификата сохраняется в свойство заказа `ORDER_ADVANCE_CERTIFICATE_PROP_CODE` для авансовых платежей
- Проверить, что оплата заказа отправляется в 1С и статус заказа устанавливается в "в обработке" после получения ответа от 1С
- Убедиться, что логика обработки сертификатов не нарушена (сохранение номера и купона, создание купона, изменение статуса)
- Проверить работу API v2 для метода `sell-certificate`

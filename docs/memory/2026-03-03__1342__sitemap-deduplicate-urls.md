# 1342: Дублирование ссылок в sitemap.xml

## Context
В сгенерированном sitemap.xml одна и та же ссылка могла встречаться несколько раз. Источники дублей: 1) каталог — у одного товара несколько торговых предложений (элементов), все они приводятся к одному URL детальной через `CatalogHelper::getProductUrl()`; 2) дополнительные ссылки (категории, фильтры, поиск) и разделы/элементы других инфоблоков могли давать URL, уже добавленные ранее; 3) форумы — при определённой логике выборки темы могли дублироваться. Поисковые системы рекомендуют не дублировать URL в sitemap.

## Причинно-следственные связи
- **Причина 1:** Итерация по элементам инфоблока с проверкой только по ID элемента (`IBLOCK_MAP[$iblockId][$arElement['ID']]`). Разные элементы (офферы) дают один и тот же `getProductUrl()` → один URL добавлялся многократно.
- **Причина 2:** Ссылки из getFilterLinks(), getSectionLinks(), getSearchLinks() и разделов/списков добавлялись без проверки по уже записанным в текущий прогон URL — возможны пересечения с URL из элементов каталога и друг с другом.
- **Решение:** Ввести единый реестр добавленных за один запуск URL (`$NS['SITEMAP_URLS']`): инициализация в шаге `init`; перед каждым вызовом `addIBlockEntry($url, ...)` проверка `isset($NS['SITEMAP_URLS'][$url])`; при отсутствии — запись в реестр и вызов addIBlockEntry.
- **Следствие:** В sitemap попадает не более одной записи на URL; объём файла и количество записей становятся предсказуемыми; выполняется рекомендация поисковиков.

## Key Decisions
- Реестр хранится в `$NS['SITEMAP_URLS']` (ассоциативный массив, ключ — URL), общий для всех шагов генерации (файлы, инфоблоки, форумы).
- Проверка выполняется во всех местах добавления URL: элементы и разделы инфоблоков, дополнительные ссылки каталога (фильтры, разделы, поиск), список инфоблока, темы форума.
- Не меняется порядок обхода и логика разбиения на части (SitemapRuntime); дедупликация только перед записью.

## Code Touchpoints
- `app/local/php_interface/lib/app/Seo/Sitemap.php`:
  - Шаг `init`: `$NS['SITEMAP_URLS'] = [];`
  - Элементы каталога (iblock): после получения `$url` через getProductUrl — `if (isset($NS['SITEMAP_URLS'][$url])) continue;` и `$NS['SITEMAP_URLS'][$url] = true;` перед addIBlockEntry.
  - Разделы (не каталог): обёртка addIBlockEntry в `if (!isset($NS['SITEMAP_URLS'][$url])) { ... }`.
  - Доп. ссылки (getFilterLinks, getSectionLinks, getSearchLinks): в цикле по `$link` проверка и установка в реестр перед addIBlockEntry.
  - Список инфоблока (LIST_PAGE_URL): та же обёртка по `$url`.
  - Форум (два места): проверка и установка перед addIBlockEntry для темы и для альтернативной ветки.

## Gotchas
- Переменная `$url` в цикле по getFilterLinks/getSectionLinks/getSearchLinks в блоках `if ($generateJson)` ранее ошибочно использовалась как `$url` (из внешнего контекста); при дедупликации в цикле корректно использовать `$link` для записи в links и в реестр.
- Реестр не сбрасывается между шагами — намеренно, чтобы один и тот же URL из разных источников не дублировался.

## Verification
- Запустить генерацию sitemap (cron или вручную: `app/local/cron/seo/generate_sitemap.php` или вызов `(new \App\Seo\Sitemap())->generate(false)`).
- Проверить сгенерированный XML: поиск по одному и тому же URL в файлах sitemap — каждое значение должно встречаться один раз.
- Сравнить количество записей до и после правки (после — не больше и обычно меньше за счёт устранения дублей).

## Follow-ups
- При добавлении новых источников URL в генерацию sitemap — перед addIBlockEntry всегда проверять и обновлять `$NS['SITEMAP_URLS']`.
